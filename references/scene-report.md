# 汇报分册（周报 / 月报 / 季度半年总结 / 述职）

## 结构硬约束

1. **首页即 dashboard**：关键结果（每条带 vs 基线）＋ 做的好的 vs 问题与挑战（逐条对称，不写单边 scorecard）＋ 下阶段计划（≤4 条，每条一行）。读者只看首屏就能拍板
2. **关键结果按目标线拆**：不是挑亮眼的事，而是每条目标线各报一条——目标 → 当前进展 → 下一步动作和时间。正文用业务名词组织，内部编号（KR1/子线）不出现
3. **好坏按结果影响力排**：挑对目标达成影响最大的 3 件好事和 3 个问题，不按维度平分；素材撑不起 3v3 时按实情取（2v2 也行），不硬凑
4. **字数**：周报 ≤800 字，月报 ≤1000 字；超了直接砍内容，不压句子

## 写作流程：先详后简

先写信息饱和的详细版（每条线含进展事实 + so what + 动作背景 + 下期计划），再概括到目标字数。压缩发生在完整思考之后——一稿压字会砍掉 so what 和数据基线，让句子电报化，这是模板腔的根源。

## 内容规则

- **挑战写因果链不写问题清单**：一句定性 + 已做了什么 + 在哪受限 + 受限的具体操作层细节
- **负面数字不藏，同句归因**：成绩与失败写进同一句，归因紧跟数字——"路径缩短 63%（16 步→6 步），但试点使用率低于 10%，核心原因是素材不足加工具使用门槛"。归因写完，有数字依据时再补正向定调
- **趋势逐拐点归因**：曲线/月度数据的每个拐点写明触发事件（"6 月渠道调整划走网点，跌到 8%；11-12 月考核口径回归，回升；1 月疫情仓配关停，跌到 7.6%"）——没有事件解释的波动等于没分析
- **未完成的数据标【未完成】**，不补虚话——坦白 work-in-progress 是信任的基础
- **每段正文至少 3 个硬数**，数不够就补数据或合并段落
- **板块开头概括句压 5 要素进 60 字**：时间范围 + 前提 + 核心数据 1-3 个 + 坦白哪里还不行
- **敏感措辞**：人员变动写"资源向高潜力新人倾斜"类含蓄表述；业绩缺口给数字 + 原因 + 追赶动作，不恐慌化也不回避。交付前扫一遍禁用词清单（任一命中需重写）：裁员、淘汰、末位、跟不上、尾部、退出、降级处理、大力度启用、直接转进、没有历史负担。可机器扫：`bash scripts/check.sh 文件.md report`（机器清单以该脚本为准，增删词时同步改脚本和上面这行）
- **口径冻结**：多源数字先对账（差异解释到分母/分子/时点），冻结唯一填报口径，旧口径登记作废；两套分组明细不进同一张表；人数类标注统计边界

## REVIEW：逐句过五关

每句话过：① 话讲完了吗（没说一半留人猜）② so what（这个事实意味着什么）③ 说人话了吗（没用抽象概念词替代事实）④ 数据有 vs 吗（vs 目标/历史/对照）⑤ 然后呢（判断闭环：好坏 + 原因 + 下一步）。任一关不过就标记重写，循环到全文零标记。
